## Summary

For approximately four weeks, every mutation through `tasks/services.py`
(task create / update / reorder / promote, collaborator add / remove)
silently failed to write an `AuditLog` row. Zero entries were produced
for the entire Helm Tasks surface during this window. Because the data
was never written, **the gap is unrecoverable** â€” there is no source
from which to backfill.

## Timeline (UTC)

- **2026-03-26** â€” commit `9fec43c` ("feat: optional Helm Tasksâ€¦")
  introduces `tasks/services.py` with a private `_audit()` helper that
  passes `target_type=`, `target_id=`, `metadata=` as kwargs to
  `core.AuditLog.objects.create(...)`.
- **2026-03-26 â†’ 2026-04-25** â€” every call raises `TypeError` because
  the actual model (concrete subclass of `keel.core.models.AbstractAuditLog`)
  uses fields `entity_type`, `entity_id`, `description`, `changes`. The
  exception is swallowed by a bare `except: pass` (lines 26â€“29) marked
  `# pragma: no cover`.
- **2026-04-25** â€” gap discovered; hotfix prepared, reviewed, deployed
  same-day.

## Root cause

Two mistakes compounded:

1. **Field-name drift between `_audit()` and the concrete model.** The
   helper was written against a remembered/older shape of `AuditLog`,
   not the current `AbstractAuditLog`. There was no integration test
   exercising the real model â€” only the helper's call signature.
2. **A bare `except: pass` swallowed the `TypeError`.** The justifying
   comment ("AuditLog field shape may differ across keel versions")
   converted a loud, immediate failure into a silent four-week gap.
   `# pragma: no cover` then suppressed it from coverage reports too.

## Fix (this PR)

- Delete `_audit()`. Replace each of the six call sites in
  `tasks/services.py` with a direct call to the canonical
  `keel.core.audit.log_audit()` helper, using `target._meta.label`
  for `entity_type` (e.g. `helm_tasks.Task`) and `str(target.pk)` for
  `entity_id`.
- Add a regression test
  (`test_create_task_writes_audit_log_entry`) that asserts a real
  `AuditLog` row is written by `create_task()` â€” the test that, had it
  existed on 2026-03-26, would have caught this on day zero.

`log_audit()` itself still catches and logs exceptions, but it is the
shared, exercised path used by the rest of the platform; failures there
will surface in `keel.core.audit` log channels rather than being
silently dropped by an app-local helper.

## Data impact

- **Window:** 2026-03-26 â†’ 2026-04-25 (~30 days).
- **Affected actions:** `task.create`, `task.update`, `task.reorder`,
  `task.promote`, `task.collaborator.add`, `task.collaborator.remove`.
- **Backfill:** not possible. Source data (timestamps, actor, change
  set, IP) was never recorded anywhere else.
- **Compliance:** documented in this post-mortem; downstream consumers
  of the audit table (compliance review, security review) should treat
  the window as "no data" rather than "zero events."
